# Q1. How is memory managed in Java?

## Answer

Java manages memory automatically through the **Java Virtual Machine (JVM)**.

The JVM divides memory into different areas, each serving a specific purpose.

```
                JVM Memory

        +----------------------+
        |      Heap Memory     |
        +----------------------+
        |      Stack Memory    |
        +----------------------+
        |   Method Area        |
        +----------------------+
        | Program Counter (PC) |
        +----------------------+
        | Native Method Stack  |
        +----------------------+
```

---

## 1. Heap Memory

Heap memory stores:

- Objects
- Instance Variables
- Arrays

All objects created using the `new` keyword are stored in the Heap.

Example

```java
Student s = new Student();
```

The object is stored in the Heap.

---

## Heap Structure

```
Heap

│

├── Young Generation
│      ├── Eden Space
│      ├── Survivor S0
│      └── Survivor S1
│
└── Old Generation
```

### Young Generation

Newly created objects are stored here.

Most objects die here.

---

### Old Generation

Objects that survive multiple Garbage Collection cycles are moved here.

---

## 2. Stack Memory

Every thread gets its own Stack.

Stack stores:

- Local Variables
- Method Calls
- References to Objects

Example

```java
public void show() {

    int age = 25;

}
```

`age` is stored in the Stack.

---

## Stack Example

```java
Student s = new Student();
```

```
Stack

s
 │
 ▼

Heap

Student Object
```

The reference is stored in the Stack.

The actual object is stored in the Heap.

---

## 3. Method Area

Stores

- Class Metadata
- Static Variables
- Static Methods
- Constant Pool

Example

```java
static int count = 10;
```

Stored in Method Area.

---

## 4. Program Counter (PC Register)

Stores the address of the currently executing JVM instruction.

Each thread has its own PC Register.

---

## 5. Native Method Stack

Used when Java code calls native methods written in languages like C or C++.

Example

JNI (Java Native Interface)

---

## Complete Memory Flow

```java
Student s = new Student();
```

```
Stack

Reference

↓

Heap

Student Object

↓

Method Area

Student Class Information
```

---

## Real-world Example

Suppose an e-commerce application creates thousands of Order objects.

```
new Order()

↓

Heap

↓

Order Processing

↓

Object becomes unreachable

↓

Garbage Collector removes it
```

---

## Interview Tip

Remember this one-liner:

```
Heap

↓

Objects
```

```
Stack

↓

Method Execution
```

```
Method Area

↓

Class Information
```

---

## Quick Revision

| Memory Area | Stores |
|-------------|--------|
| Heap | Objects |
| Stack | Local Variables & References |
| Method Area | Class Metadata & Static Members |
| PC Register | Current Instruction |
| Native Stack | Native Methods |

---

## Common Mistakes

❌ Objects are stored in the Stack.

❌ Static variables are stored in Heap.

❌ Every thread shares the same Stack.

# Q2. How does Garbage Collection work?

## Answer

Garbage Collection (GC) is an automatic memory management process in Java.

It identifies objects that are no longer reachable and removes them from the Heap memory.

This helps prevent memory leaks and frees unused memory automatically.

---

## How GC Works

```
Object Created

↓

Heap

↓

Object No Longer Referenced

↓

Eligible for Garbage Collection

↓

Garbage Collector Removes It
```

---

## Example

```java
Student s = new Student();

s = null;
```

The object has no reference.

It becomes eligible for Garbage Collection.

---

## Another Example

```java
public void test() {

    Student s = new Student();

}
```

After the method completes,

the reference is removed from the Stack,

making the object eligible for GC.

---

## Is GC Immediate?

No.

Garbage Collection is **not guaranteed** to run immediately.

The JVM decides when to run it.

---

## Can We Force GC?

We can only request it.

```java
System.gc();
```

or

```java
Runtime.getRuntime().gc();
```

The JVM may ignore the request.

---

## Types of Garbage Collection

### Minor GC

Runs in the Young Generation.

Fast.

Occurs frequently.

---

### Major GC

Runs in the Old Generation.

Slower than Minor GC.

---

### Full GC

Cleans the entire Heap.

Includes:

- Young Generation
- Old Generation

Most expensive.

---

## Object Lifecycle

```
new Object()

↓

Young Generation

↓

Survives GC

↓

Old Generation

↓

No References

↓

Garbage Collected
```

---

## Real-world Example

Suppose an API creates thousands of request objects.

```
Request Arrives

↓

Object Created

↓

Response Sent

↓

Object Becomes Unreachable

↓

GC Removes It
```

---

## Interview Tip

Don't say

```
GC destroys unused objects.
```

Say

```
GC removes unreachable objects.
```

That is the correct interview terminology.

---

## Advantages

- Automatic Memory Management
- Prevents Memory Leaks
- Frees Heap Memory
- Improves Application Stability

---

```
Object

↓

Unreachable

↓

```

❌ Calling `System.gc()` guarantees GC.

❌ GC removes objects immediately.

❌ GC removes objects that are still referenced.

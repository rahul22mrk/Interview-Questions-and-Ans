# Q1. What is the difference between remove(int index) and remove(Object o)?

## Answer

The `remove()` method is overloaded in Java Collections.

### 1. remove(int index)

Removes the element present at the specified index.

Returns the removed element.

Example

```java
List<String> list = new ArrayList<>();

list.add("Java");
list.add("Spring");
list.add("SQL");

System.out.println(list.remove(1));

System.out.println(list);
```

Output

```
Spring

[Java, SQL]
```

---

### 2. remove(Object o)

Removes the first occurrence of the specified object.

Returns `true` if the object is removed.

Returns `false` if the object is not found.

Example

```java
List<String> list = new ArrayList<>();

list.add("Java");
list.add("Spring");
list.add("SQL");

System.out.println(list.remove("Spring"));

System.out.println(list);
```

Output

```
true

[Java, SQL]
```

---

## Integer Example (Very Important Interview Question)

```java
List<Integer> list = new ArrayList<>();

list.add(10);
list.add(20);
list.add(30);

list.remove(1);

System.out.println(list);
```

Output

```
[10, 30]
```

Because `1` is treated as an **index**.

---

If you want to remove the value `20`:

```java
list.remove(Integer.valueOf(20));
```

Output

```
[10, 30]
```

---

## Difference

| remove(int index) | remove(Object o) |
|-------------------|------------------|
| Removes by index | Removes by value |
| Returns removed element | Returns boolean |
| Throws IndexOutOfBoundsException if index is invalid | Returns false if object is not found |

---

## Real-world Example

Suppose an e-commerce application stores products in a cart.

- Remove by position → `remove(index)`
- Remove by product → `remove(product)`

---

## Quick Revision

```text
remove(index)  → Remove using index

remove(object) → Remove using value
```

---

## Common Mistakes

❌ Thinking `remove(1)` removes the value `1`.

For `List<Integer>`, it removes the element at index `1`.

---

# Q2. What happens if you iterate over a collection while removing elements from it?

## Answer

It depends on how you remove the element.

If you modify the collection directly while iterating using a for-each loop or Iterator, Java throws a

```
ConcurrentModificationException
```

---

## Wrong Example

```java
List<String> list = new ArrayList<>();

list.add("Java");
list.add("Spring");
list.add("SQL");

for (String s : list) {

    if (s.equals("Spring")) {
        list.remove(s);
    }

}
```

Output

```
ConcurrentModificationException
```

---

## Why?

Because the Iterator detects that the collection has been modified outside of its control.

---

## Correct Way

Use Iterator.

```java
Iterator<String> iterator = list.iterator();

while (iterator.hasNext()) {

    String value = iterator.next();

    if (value.equals("Spring")) {
        iterator.remove();
    }

}
```

No exception.

---

## Java 8 Way

```java
list.removeIf(name -> name.equals("Spring"));
```

---

## HashMap vs Hashtable vs ConcurrentHashMap

### HashMap

Not thread-safe.

Iterator is fail-fast.

Throws

```
ConcurrentModificationException
```

---

### Hashtable

Thread-safe.

Still uses fail-fast Iterator.

Can throw

```
ConcurrentModificationException
```

---

### ConcurrentHashMap

Thread-safe.

Iterator is fail-safe.

No ConcurrentModificationException during iteration.

Allows modification while iterating.

Example

```java
ConcurrentHashMap<Integer, String> map =
        new ConcurrentHashMap<>();

map.put(1, "Java");
map.put(2, "Spring");

for (Integer key : map.keySet()) {

    map.put(3, "SQL");

}
```

No exception.

---

## Real-world Example

ConcurrentHashMap is commonly used in

- Caching
- Session Management
- Multi-threaded Applications

---

## Quick Revision

```text
HashMap

↓

Fail-Fast

↓

ConcurrentModificationException
```

```text
ConcurrentHashMap

↓

Fail-Safe

↓

No Exception
```

---

## Common Mistakes

❌ HashMap is thread-safe.

❌ ConcurrentHashMap locks the whole map.

(It locks only a portion internally.)

---

# Q3. What happens if you try to add an element to an immutable list created using List.of() or a fixed-size list created using Arrays.asList()?

## Answer

Both `List.of()` and `Arrays.asList()` restrict modifications, but in different ways.

---

## List.of()

Creates a completely immutable list.

No modification is allowed.

Example

```java
List<String> list = List.of(
        "Java",
        "Spring"
);

list.add("SQL");
```

Output

```
UnsupportedOperationException
```

---

## Arrays.asList()

Creates a fixed-size list backed by an array.

You can update existing elements.

You cannot change the size.

Example

```java
List<String> list =
        Arrays.asList(
                "Java",
                "Spring"
        );

list.add("SQL");
```

Output

```
UnsupportedOperationException
```

---

But

```java
list.set(0, "Python");
```

Works.

Output

```
[Python, Spring]
```

---

## Difference

| List.of() | Arrays.asList() |
|------------|-----------------|
| Immutable | Fixed-size |
| add() ❌ | add() ❌ |
| remove() ❌ | remove() ❌ |
| set() ❌ | set() ✅ |

---

## Real-world Example

`List.of()` is useful when configuration values should never change.

`Arrays.asList()` is useful when you need a fixed-size list backed by an array.

---

## Quick Revision

```text
List.of()

↓

Immutable
```

```text
Arrays.asList()

↓

Fixed Size
```

---

## Common Mistakes

❌ Arrays.asList() returns ArrayList.

(It returns a fixed-size List backed by an array.)

❌ List.of() allows set().

(It doesn't.)
